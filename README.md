# ECMAScript Environment Compatibility Table

A simplified ECMAScript compatibility table for developers who want a quick way to compare JavaScript language features against common runtimes.

https://soul-master.github.io/ecma-env-compat-table/

This project is designed to help when choosing TypeScript `target` and `lib` settings. Instead of reading many separate compatibility pages, you can open one table, expand an ECMAScript edition, and see which browser or Node.js version is needed for the features in that edition.

## What This Repo Contains

- `index.html` - the static web page and styles.
- `src/index.mjs` - the browser-side JavaScript that loads compatibility data and renders the table.
- `data/edition-mapping.json` - the ECMAScript editions, feature groups, descriptions, and MDN Browser Compat Data paths used by the table.

There is no build step and no package install required.

## What The Table Shows

The app groups JavaScript features by ECMAScript edition, such as ES2016, ES2020, ES2024, and newer candidate editions.

Each edition can be expanded to show individual features. For each feature, the table shows support information for:

- Chrome / Edge
- Safari
- Firefox
- Node.js

Parent edition rows summarize the highest minimum version required by the visible child features. For example, if one ES2022 feature requires a newer browser version than the others, the ES2022 row will show that newer required version.

## Why This Helps With TypeScript

TypeScript has two important compiler options that often confuse beginners:

- `target` controls what JavaScript syntax TypeScript outputs.
- `lib` controls which built-in JavaScript and web APIs TypeScript assumes are available.

For example:

```json
{
  "compilerOptions": {
    "target": "ES2020",
    "lib": ["ES2020", "DOM"]
  }
}
```

This table helps you understand what features belong to each ECMAScript edition and whether your target browsers or Node.js version support them.

For commercial grade products, browser support is often defined by a policy instead of a fixed browser list. For example, a team might support only the last 3 major versions of specific browsers. That support window changes every year, especially for Safari because new major Safari versions are tied closely to Apple platform releases.

Because the supported browser set changes over time, your TypeScript `target` and `lib` settings should be reviewed again regularly. A config that was correct last year may be too old or too new for this year's official support policy.

Important: this repo does not automatically generate a `tsconfig.json` for you. It is a reference tool to help you make a more informed `target` and `lib` choice.

## Data Sources

Runtime compatibility data is loaded directly in the browser from the current `@mdn/browser-compat-data` package on the Unpkg CDN:

```text
https://unpkg.com/@mdn/browser-compat-data/data.json
```

That means compatibility results can change over time as MDN Browser Compat Data is updated.

The ECMAScript edition feature list in `data/edition-mapping.json` was generated by ChatGPT 5.5, then mapped to MDN Browser Compat Data paths used by the app.

## How To Run

Because the app uses browser modules and fetches compatibility data from a CDN, run it from a local static server.

One simple option is `http-server`:

```sh
npx http-server
```

Then open the local URL shown by the command. By default, it is usually:

```text
http://localhost:8000
```

## How It Works

1. The browser loads `data/edition-mapping.json`.
2. The app downloads the latest MDN Browser Compat Data from Unpkg.
3. Each configured feature path is resolved inside the MDN data.
4. Support versions are summarized for each runtime column.
5. The table renders editions as expandable parent rows with feature rows underneath.

Unresolved MDN data paths are shown as `Unknown`. You can hide or show those rows with the checkbox in the toolbar.

## Limitations

- Compatibility data depends on the latest CDN copy of `@mdn/browser-compat-data`.
- Some ECMAScript features do not map cleanly to a single MDN Browser Compat Data path.
- The table is a simplified guide, not a complete replacement for reading TypeScript, TC39, runtime, or MDN documentation.
- TypeScript `target` and `lib` choices may also depend on bundlers, transpilers, polyfills, and your actual deployment environment.
